AGENT ARCHITECTURE — DEFINE THESE AGENTS IN ANTIGRAVITY
==========================================================================

Spawn and orchestrate the following named agents. Each agent has a 
specific responsibility. Antigravity must show the handoff between agents 
in its trace logs.

AGENT 1 — INTENT_AGENT
  Role: Parse multilingual input (Urdu, Roman Urdu, English, code-switched)
  Tools: Claude API (claude-sonnet-4-20250514), Gemini Flash
  Output schema:
    {
      "service_type": string,
      "location": string,
      "time_preference": string,
      "urgency_level": "low|medium|high|emergency",
      "budget_sensitivity": "low|medium|high",
      "language_detected": string,
      "confidence_score": float (0.0-1.0),
      "clarification_needed": boolean,
      "clarification_question": string | null,
      "raw_input": string
    }
  Behavior: If confidence_score < 0.75, set clarification_needed = true 
  and generate a clarification question in the same language as input.
  Handle misspellings, slang, incomplete sentences.

AGENT 2 — PROVIDER_DISCOVERY_AGENT
  Role: Find relevant providers from mock dataset + Google Places API
  Tools: Google Maps/Places API, Firebase Firestore (providers collection)
  Input: service_type, location (lat/lng resolved via Geocoding API)
  Output: List of up to 10 raw provider candidates with:
    id, name, service_types[], location, lat, lng, distance_km, 
    availability_slots[], base_rate_pkr, rating, review_count, 
    on_time_score, cancellation_rate, specialization_tags[], 
    last_active, capacity_today, risk_score

AGENT 3 — MATCHING_AGENT
  Role: Rank providers using multi-factor scoring algorithm
  Tools: Claude API for reasoning, internal scoring function
  Scoring formula (weighted, total = 100):
    - Distance/travel time         : 20 points
    - Availability match           : 20 points
    - Rating + review recency      : 15 points
    - On-time / reliability score  : 15 points
    - Skill specialization match   : 10 points
    - Price vs user budget         : 10 points
    - Cancellation risk (inverse)  : 5 points
    - User preference history      : 5 points
  Output: Ranked list (top 3) with per-factor breakdown + human-readable 
  reasoning string explaining WHY the top provider was chosen.
  IMPORTANT: Reasoning must be transparent. Example: "Provider A ranked 
  #1 despite being 0.8km farther than Provider B because Provider A has 
  a 94% on-time score vs Provider B's 61% and specializes in inverter AC 
  repair which matches the detected job complexity."

AGENT 4 — PRICING_AGENT
  Role: Generate dynamic price quote with full breakdown
  Tools: Claude API, internal pricing engine
  Inputs: service_type, urgency_level, distance_km, base_rate_pkr, 
          job_complexity, budget_sensitivity, time_of_day, demand_index
  Pricing formula:
    base_fee = provider.base_rate_pkr
    distance_fee = distance_km * 15 (PKR per km)
    urgency_multiplier = { low: 1.0, medium: 1.15, high: 1.3, emergency: 1.6 }
    complexity_multiplier = { basic: 1.0, intermediate: 1.25, complex: 1.5 }
    surge_multiplier = demand_index > 0.8 ? 1.2 : 1.0
    loyalty_discount = returning_user ? 0.95 : 1.0
    total = base_fee + distance_fee * urgency * complexity * surge * loyalty
  Output:
    {
      "base_fee": number,
      "distance_fee": number,
      "urgency_adjustment": number,
      "complexity_adjustment": number,
      "surge_fee": number,
      "loyalty_discount": number,
      "total_estimate_pkr": number,
      "quote_range": { "min": number, "max": number },
      "budget_alternative": object | null,
      "breakdown_text": string (in user's language)
    }
  If budget_sensitivity = high, always include a budget_alternative 
  with a lower-cost provider option.

AGENT 5 — BOOKING_AGENT
  Role: Simulate full booking confirmation workflow
  Tools: Firebase Firestore (bookings collection), Firebase Functions, 
         mock SMS/WhatsApp notification service
  Actions to execute (ALL must be logged in Antigravity trace):
    1. Check slot availability (prevent double-booking)
    2. Reserve slot in provider's calendar
    3. Write booking record to Firestore:
       { booking_id, user_id, provider_id, service_type, slot_datetime,
         location, price_quote, status: "confirmed", created_at }
    4. Generate booking confirmation message (in user's language)
    5. Simulate WhatsApp/SMS notification to user
    6. Simulate notification to provider
    7. Generate PDF/text booking receipt
    8. Schedule reminder job (T-60 minutes before appointment)
  Output: booking_id, confirmation message, receipt text, 
          simulated notification log

AGENT 6 — FOLLOWUP_AGENT
  Role: Manage post-booking lifecycle
  Tools: Firebase Functions (scheduled triggers), Firestore
  Workflow:
    T-60min : Send reminder to user ("Your AC technician arrives in 1 hour")
    T-0     : Mark service as "in_progress", simulate provider en-route update
    T+service_duration : Mark as "completed", trigger feedback request
    Feedback collection: 1-5 star rating + optional text comment
    Post-feedback: Update provider's rating (moving average), 
                   adjust risk_score and future matching weight
  Output: Updated provider reputation object, feedback receipt

AGENT 7 — DISPUTE_AGENT
  Role: Handle complaints, no-shows, cancellations, refunds, escalation
  Tools: Claude API for classification, Firestore (disputes collection)
  Trigger conditions and responses:
    no_show          → Auto-reschedule + waive rebooking fee + log incident
    cancellation_<2h → Apply cancellation penalty, find replacement provider
    quality_complaint→ Collect evidence (photo placeholder), freeze rating, 
                       escalate to human review queue
    price_dispute    → Show original quote vs charged amount, offer partial 
                       refund if overcharged by >10%
    blacklist        → If provider has 3+ disputes in 30 days, flag account
  Output: dispute_id, resolution_action, user_message, escalation_flag

==========================================================================
ANTIGRAVITY WORKFLOW DEFINITION
==========================================================================

Define this exact workflow in Antigravity's planner:

WORKFLOW: service_request_lifecycle
  
  STEP 1 [PLAN]:
    Agent: INTENT_AGENT
    Action: parse_multilingual_input(user_input)
    On confidence < 0.75: pause, send clarification, await user response
    On success: emit intent_object → STEP 2

  STEP 2 [DISCOVER]:
    Agent: PROVIDER_DISCOVERY_AGENT  
    Action: discover_providers(intent.service_type, intent.location)
    On no providers found: notify user, suggest alternate time/location
    On success: emit raw_provider_list → STEP 3

  STEP 3 [MATCH]:
    Agent: MATCHING_AGENT
    Action: rank_providers(raw_provider_list, intent)
    On tie: prefer higher on_time_score
    On all low scores (<40): warn user, show anyway with caveats
    On success: emit ranked_list → STEP 4 (parallel)

  STEP 4 [PRICE — runs parallel with STEP 3 result]:
    Agent: PRICING_AGENT
    Action: generate_quote(top_provider, intent)
    On budget conflict: generate_budget_alternative()
    On success: emit price_quote → STEP 5

  STEP 5 [DECIDE]:
    Antigravity synthesizes ranked_list + price_quote
    Constructs recommendation payload
    Presents to user for confirmation
    Awaits user action: [Book Now] or [See Other Options]

  STEP 6 [BOOK]:
    Agent: BOOKING_AGENT
    Action: create_booking(confirmed_provider, slot, quote)
    On slot conflict: suggest_next_available_slot()
    On success: emit booking_confirmation → STEP 7

  STEP 7 [FOLLOWUP — async]:
    Agent: FOLLOWUP_AGENT
    Action: schedule_lifecycle(booking_id)
    Runs independently via Firebase scheduled functions
    Updates Firestore booking status at each milestone

  ERROR HANDLING AT EVERY STEP:
    Log error to agent_logs with step name, error type, input snapshot
    Attempt retry once with exponential backoff
    On second failure: graceful degradation + user notification
    Always maintain session continuity
